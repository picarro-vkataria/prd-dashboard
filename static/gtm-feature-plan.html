Staffing analysis based on SP volume, FE/BE split by milestone, and domain specialization requirements. These risks assume a standard team of 2–3 FE + 2–3 BE engineers. Velocity assumption: ~30–35 SP/engineer/milestone at standard pace (1.0×).
Critical MS6 + MS8 Are 2× Average Milestone Load
MS6 = 131 SP and MS8 = 128 SP — nearly double the average milestone (85 SP). MS6 alone requires 3 pipelines to ship simultaneously: AutoFOV display, Gaps pipeline, and LISA pipeline. MS8 adds hardware ingest (H-01/H-02 = 24 BE SP) on top of User Management and Staging environments.
Impact: At standard 30 SP/engineer/milestone, MS6 needs at least 4–5 engineers (2 FE + 2–3 BE). If the team is sized for average load (~3 engineers), MS6 and MS8 will overrun by 30–50%.
Mitigation: Rebalance scope from MS6 into MS5 (pre-ship BM prep) and MS7 (post-ship cleanup). Or staff up to 5 during MS6 window.
Critical Hardware Has Zero Jira Epics & Zero Owners
All 11 hardware features (94 SP total) are unplanned in Jira. H-00 (Handheld ingest, 13 SP) and H-01 (Car ingest, 13 SP) are each 2 FE / 11 BE — the BE work requires hardware SDK integration and firmware communication protocols. Hardware procurement + firmware development typically requires 8–12 week lead time. MS8 is when hardware features start — every week of delay in assigning a hardware lead today = one week of risk at MS8.
Impact: If no hardware engineer is assigned by MS5 start, H-01/H-02 will not be ready for MS8.
Mitigation: Assign hardware lead now. Create Jira epics for H-00 through H-13. Firmware integration contract must be established before MS6.
Critical Mobile Requires Dedicated Mobile FE (Not Web FE)
The 5 mobile features total 60 SP at 70% FE (42 FE SP). M-DV-02 (Driver View Capture Mode) alone is 21 SP / 15 FE SP — the single largest FE feature in the entire plan. It requires real-time indication capture, GPS tracking, and works on ChromeOS and iPadOS. A web FE engineer assigned to mobile will take 2× longer and produce lower quality. M-DV-01 starts in MS7 — a mobile FE engineer needs to be onboarded by that milestone.
Impact: Using a web FE engineer for mobile = ~30–42 SP effectively becomes 60+ SP in execution time.
Mitigation: Hire or contract a mobile FE specialist before MS7. Platform decision (browser vs native iOS/Android) must be locked in MS5–MS6 (Apple App Store review = 4–8 weeks if iOS).
High P-07 Security & Compliance Has No Milestone and No Owner
P-07 (GDPR / ISO 27001 / SOC 2) is 13 SP, 12 BE SP, currently unscheduled (MS?). This is a ship blocker for EU customers — EU product launch requires GDPR compliance. SOC 2 audits typically require 6+ months of preparation including logging, access controls, and change management evidence. The audit process itself cannot start until the platform is production-stable.
Impact: If P-07 is not placed in a milestone by MS7, EU launch (planned MS11) is at risk.
Mitigation: Schedule P-07 in MS7 or MS8. Assign a security-focused BE engineer. Begin SOC 2 readiness checklist now.
High MS5 is 71% BE — FE Engineers Idle in Early Milestones
MS4 = 45% FE, MS5 = only 29% FE (19 SP). The first two milestones are dominated by platform infra (auth, GIS pipelines, staging, logging) which is almost entirely BE work. A team with 2 FE + 2 BE will have FE engineers significantly underutilized in MS4–MS5, then suddenly overloaded in MS6 (50% FE, 66 SP).
Impact: FE engineers may be blocked or underutilized for 2 milestones, then face a spike in MS6.
Mitigation: Use MS4–MS5 FE time for design system work, component library, Figma-to-code prep, and test infrastructure. Or re-sequence some MS6 FE features (e.g. Campaign CRUD, admin screens) into MS5.
High P-18 Data Migration is High-Risk and Unowned
P-18 (Historical Data Migration, P-Cubed → PCP) is 13 SP, 12 BE SP, scheduled in MS9. Data migrations are notoriously underestimated. The P-Cubed data model may have inconsistencies, edge cases, and customer-specific configurations that only surface during migration. A failed or partial migration in MS9 could block MS10 and MS11 delivery.
Impact: If migration fails, customer onboarding to PCP is blocked regardless of feature completeness.
Mitigation: Assign dedicated BE to P-18 in MS8 for investigation/prototyping (treat as 2-milestone effort). Run migration against a staging environment first. Build rollback procedure before starting production migration.
Medium 3 Data Pipelines (AutoFOV + Gaps + LISA) Share a Single Critical Path
O-BM-12 (AutoFOV, 3FE/10BE), O-BM-17 (Gaps, 3FE/10BE), and O-BM-24 (LISA, 3FE/10BE) are each 13 SP BE-heavy pipelines. All 3 land in MS6. AutoFOV is the upstream dependency: Gaps and LISA both consume AutoFOV output. If AutoFOV pipeline (O-BM-12) delays, all downstream Gaps and LISA work is blocked. This is compounded by the fact that the LISA pipeline has zero technical design documented today.
Impact: A 2-week AutoFOV delay cascades to both Gaps and LISA, potentially pushing them to MS7.
Mitigation: Complete LISA microservice TDD before MS6 starts. Treat O-BM-12 as the MS6 "anchor" — deliver it in Sprint 1. Assign one dedicated pipeline BE engineer who owns all three.
Medium Observer View (H-05) Requires Real-Time Streaming Infrastructure
H-05 (Observer View — live view from the car) is estimated at 13 SP (8 FE / 5 BE). This is FE-heavy because it requires a real-time streaming UI (WebSocket or WebRTC), but the BE must also support low-latency telemetry streaming from the car hardware. This is architecturally different from all other features — it cannot share standard REST API patterns.
Impact: If the real-time infrastructure is not designed early, H-05 in MS10 will block the observer workflow.
Mitigation: Architecture decision for streaming (WebSocket vs SSE vs WebRTC) must be made in MS7. Prototype in MS8 alongside H-01/H-02 hardware work.
Insight Boundary Management is the Core of the Product (34% of All SP)
At 232 SP across 36 features, Boundary Management is by far the largest group — 34% of total product SP. It spans MS5–MS11. The good news: it's nearly balanced at 51% FE / 48% BE, which means a standard FE+BE pair can co-own it across milestones without significant imbalance. The 3 pipeline features (AutoFOV/Gaps/LISA) account for 39 BE SP total but unlock 33 display features worth 193 SP.
Recommendation: Assign a dedicated BM "pod" (1 FE + 1 BE) from MS5 through MS11. This team owns the full BM vertical slice and can build institutional knowledge that prevents context-switching overhead.
Insight Administration Group is FE-Dominant — Good Early FE Ramp Work
The Administration group (18 features, 82 SP) is 58% FE and mostly in MS4–MS6. These are largely CRUD admin forms and settings panels — relatively low BE complexity (mostly standard API endpoints). This makes them ideal work to assign FE engineers during the early milestones when BE is dominated by platform/infra.
Recommendation: In MS4–MS5 when BE is heavily loaded, assign FE engineers to Admin CRUD work (A-T-*, A-O-*, A-U-*). This eliminates FE idle time and pre-delivers admin features ahead of their milestone target.
Insight EU Compliance Gate Requires Gaps + AutoFOV — Both Are MS6 BE Features
EU Methane Regulation 2024/1787 requires Gaps detection and FOV operational for LDAR compliance certification. O-BM-12 (AutoFOV pipeline, MS5/MS6) and O-BM-17 (Gaps pipeline, MS6) are the two features that unlock EU customer onboarding. Both are ~80% BE SP. EU customers cannot be onboarded until these ship.
Recommendation: Treat EU compliance as a formal milestone gate at MS6 end. If either pipeline slips to MS7, EU onboarding must be delayed. Track these as P0 items in every sprint.
Insight Intelligence (I-DE-01) Is a New Analytics Domain — Scope Risk in MS11
I-DE-01 (Data Explorer: Network & Program KPIs) is the only feature in the Intelligence group, scheduled at MS11. It's estimated at 13 SP (7 FE / 6 BE) — but this is a new analytics domain with no prior art in the codebase. "Network & Program level KPIs" implies aggregating data across campaigns, operators, and tenants — which requires a new data model and query layer. Placing it in MS11 (the final milestone before GA) is risky if it expands in scope.
Recommendation: Define the exact KPI set and data sources for I-DE-01 before MS9. If scope exceeds 13 SP after definition, move to V2 or a post-GA release. Do not let an undefined analytics feature block MS11 launch.

P-18 is 61 BE SP — the largest single feature in the plan. It's assigned as a parallel dedicated BE track (Sprint 4–7). If legacy data quality issues surface mid-migration, it blocks the testing freeze (Sep 1) since migrated data is needed for golden data validation.
Mitigation: Start migration analysis and data profiling in Sprint 3 (Jun). Lock data cleansing rules before Sprint 6 begins (Jul 1). Allocate 1 dedicated Avengers BE dev. Do not share with investigation work.
HIGH Driver View Platform Decision Still TBD

HIGH Hardware Lead Time — Car + Handheld Systems

If hardware is not ordered by Apr 1, firmware development (H-01/H-02) cannot start Sprint 2, pushing them to R2 and creating a cascade into Observer View (H-05) and fleet features.

HIGH FE Single Point of Failure

Mitigation: From Sprint 2: Gladiators UI dev cross-trains on BM Leaflet/map patterns alongside Avengers UI dev. In Sprint 4+, Gladiators UI can absorb BM display work if Avengers UI is blocked. Fullstack dev also cross-trained on FE from Sprint 3.
MEDIUM LISA Microservice Has No Technical Design at Kickoff
O-BM-24 (LISA pipeline, 13 SP) is in R2 Sprint 4. At the original plan kickoff the LISA microservice had zero TDD. The LISA Boundary Results PRD was only published Mar 20, 2026 and cannot be groomed before Sprint 4. Without TDD, Avengers BE cannot start the LISA pipeline.
Mitigation: Assign LISA TDD writing in R1 Sprint 2–3 (alongside AutoFOV/Gaps). TDD must be reviewed and approved before Sprint 4 start. Block Sprint 4 LISA start if TDD is not in "Reviewed" state.
